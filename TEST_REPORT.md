## Тест 1: ⚖️ Судья — «Напиши is_palindrome»

### Задача
Написать Python-функцию проверки строки на палиндром (игнорировать пробелы, регистр, знаки препинания).

### Решения агентов

| Агент | Код | Подход |
|-------|-----|--------|
| **DeepSeek** | `''.join(ch for ch in s if ch not in string.whitespace + string.punctuation).lower()` | string.whitespace |
| **MiMo V2.5** | `re.sub(r'[^a-zA-Z0-9]', '', s).lower()` | Regex |
| **Big Pickle** | `[c.lower() for c in s if c.isalnum()]` | isalnum() |
| North Mini | `''.join(c.lower() for c in s if c.isalnum())` | isalnum() + join |

### Вердикт арбитра

**Выбрано: Big Pickle** (с микро-оптимизацией от DeepSeek)

```python
def is_palindrome(s: str) -> bool:
    cleaned = "".join(c.lower() for c in s if c.isalnum())
    return cleaned == cleaned[::-1]
```

**Обоснование:** `isalnum()` — встроенный метод, нет риска ReDoS (в отличие от regex), не зависит от `string`-констант. Join вместо list — микро-оптимизация производительности.

### Вывод
Для простой задачи diversity отсутствует при 2 моделях, но появляется при 3. Арбитр выбирает оптимальный вариант.

## Тест 4: ⚖️ Судья — «Архитектура системы тегов»

### Задача
Спроектировать архитектуру Python-приложения для системы тегов.

### Решения (3 модели)

| Агент | Подход | Ключевая идея |
|-------|--------|---------------|
| **DeepSeek** | Polymorphic Tagging | `(tag_id, taggable_id, taggable_type)` |
| **MiMo V2.5** | Классическая M2M | Слои: schemas → services → repos → db |
| **Big Pickle** | Polymorphic + сервисы | `set_tags`, `find_by_tags(any/all)` |

### Вердикт арбитра

**Гибрид B + C** — классическая M2M (безопаснее) + сервисный слой Big Pickle:

```python
# Итоговая архитектура
schemas/          # Pydantic: TagCreate, TagRead
repositories/     # TagRepo, TaggingRepo
services/
  tag_service.py       # create, rename, merge
  tagging_service.py   # set_tags, get_tags
models/             # Tag, Tagging (M2M с entity_type ENUM)
```

**Ключевое улучшение от арбитра:** `entity_type` — ENUM/CHECK в БД (allowlist), а не строка. Это безопаснее.

## Анализ моделей

| Модель | Сильные стороны | Слабые стороны | Роль в ensemble |
|--------|----------------|----------------|-----------------|
| **DeepSeek V4 Flash** | Быстрая, безопасный стиль, системность | Краткие ответы | Агент A |
| **Xiaomi MiMo V2.5** | Детальные ответы, архитектурное мышление | Может быть многословна | Агент B |
| **Big Pickle** | Креативность, pipeline-мышление | Иногда таймауты | Агент C |
| **Nemotron-3 Ultra** | Глубокий анализ, синтез, безопасность | Медленная (550B) | 🏆 Арбитр |
| **North Mini Code** | Лёгкая, быстрая | Низкий diversity | Резерв |

## Выводы

1. **3 модели — оптимум**: 2 дают мало diversity, 4 — diminishing returns
2. **Параллельность**: 3 агента одновременно (лимит Hermes) → ~3-4 мин
3. **Nemotron-3 Ultra** — отличный арбитр: выбирает обоснованно, синтезирует гибриды, указывает на безопасность
4. **OpenCode Zen** — стабильный бесплатный провайдер, но возможны таймауты при большой нагрузке
5. **Режимы арбитра** — все три работают и дают разные результаты. Выбор режима критичен для качества
