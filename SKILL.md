# LLM Ensemble — ансамбль моделей с арбитром

## Overview

Одна задача → **N независимых LLM** → каждая даёт своё решение → **арбитр** выбирает или объединяет лучшее. Паттерн «Mixture of Agents» (MoA), адаптированный для Hermes.

**Зачем:** разные модели видят задачу по-разному. Одна может упустить деталь, другая — найти элегантное решение. Арбитр собирает лучшее из всех.

## When to Use

- **Критически важные задачи** — код для production, архитектурные решения, анализ безопасности
- **Творческие задачи** — нужен лучший вариант из нескольких
- **Сложные задачи** — ни одна модель не даёт гарантированно правильного ответа
- **Проверка качества** — хотите убедиться, что решение всестороннее
- **Don't use for:** простые односложные ответы, перевод, быстрые справки — оверхед не оправдан

## Как это работает

```
Пользователь: "Напиши парсер CSV на Python"
        │
        ▼
   ┌─────────────────┐
   │   ДИСПЕТЧЕР      │  Рассылает задачу N агентам
   │   (этот скил)    │  Каждый получает полный контекст
   └────────┬────────┘
            │
     ┌──────┼──────┐
     ▼      ▼      ▼
┌────────┐┌────────┐┌────────┐
│Agent A ││Agent B ││Agent C │  Каждый решает независимо
│model X ││model Y ││model Z │  (параллельно, через
└───┬────┘└───┬────┘└───┬────┘   delegate_task)
    │        │        │
    ▼        ▼        ▼
   Решение1  Решение2  Решение3
    │        │        │
    └────────┼────────┘
             ▼
   ┌─────────────────┐
   │    АРБИТР        │  Оценивает все решения
   │   (ваша модель)   │  Выбирает лучшее ИЛИ
   └────────┬────────┘  синтезирует гибрид
            │
            ▼
   ┌─────────────────┐
   │  ОПТИМАЛЬНОЕ     │  Одно решение + пояснение
   │    РЕШЕНИЕ       │  почему выбрано именно оно
   └─────────────────┘
```

## Процедура

### Шаг 1 — Определить задачу и количество агентов

Задача должна быть чётко сформулирована. Количество агентов: **2-3** (максимум для параллельного `delegate_task`).

### Шаг 2 — Выбрать модели для агентов

Все модели — через провайдер **OpenCode Zen** (бесплатно, без API-ключа, работает из коробки). Полный список: `opencode models opencode`.

**Доступные модели:**

| Модель | Настоящее имя | Вес | Спецификация |
|--------|--------------|------|-------------|
| `opencode/nemotron-3-ultra-free` | NVIDIA Nemotron-3 Ultra | **550B** | 🏆 арбитр (основной), 1M контекст |
| `opencode/deepseek-v4-flash-free` | DeepSeek V4 Flash | **284B** | 🔄 арбитр (fallback), агент A |
| `opencode/big-pickle` | Big Pickle | **70B** | Сильная архитектурная мысль |
| `opencode/mimo-v2.5-free` | Xiaomi MiMo V2.5 | **7B** | Детальные ответы |
| `opencode/north-mini-code-free` | North Mini Code | ~1-3B | Лёгкая, для простых задач |

**Тестирование показало (реальные замеры):**

| Конфигурация | Diversity | Время | Вердикт |
|-------------|-----------|-------|---------|
| 🆓 **2 модели** | Низкий для простых задач | ~2 мин | Для базовых задач |
| 🚀 **3 модели** 🔥 | **Максимальный** | ~3.5 мин | ⭐ **РЕКОМЕНДУЕТСЯ** |
| 4 модели | Есть diminishing returns | ~5 мин | Избыточно |

**Рекомендуемая комбинация (3 модели + арбитр):**

| Роль | Модель | Вес | Обоснование |
|------|--------|------|-------------|
| 🏆 **Арбитр** | `opencode/nemotron-3-ultra-free` | 550B | Лучшая для оценки и синтеза |
| 🔄 **Fallback арбитр** | `opencode/deepseek-v4-flash-free` | 284B | Если Ultra недоступен/таймаут |
| 🤖 **Агент A** | `opencode/deepseek-v4-flash-free` | 284B | Быстрая, безопасный стиль |
| 🤖 **Агент B** | `opencode/mimo-v2.5-free` (MiMo V2.5) | 7B | Детальные ответы, архитектура |
| 🤖 **Агент C** | `opencode/big-pickle` | 70B | Сильная архитектурная мысль, pipeline |

> **Для простых задач** (менее 50 слов) — ensemble не запускать, достаточно одного агента.
> **Для критических задач** — 3 модели + арбитр обеспечивают максимум diversity.
> **О нестабильности Nemotron-3 Ultra:** на бесплатном tier OpenCode Zen может падать на длинных промптах (>200 слов). В этом случае арбитром автоматически выступает DeepSeek V4 Flash (284B, fallback).
> Результаты тестов: `references/test-results-2026-06-16.md`

### Шаг 3 — Запустить параллельное решение

```python
task = "Напиши Python-функцию, которая парсит CSV с поддержкой разных разделителей и кодировок"

results = delegate_task(tasks=[
    {
        "goal": task,
        "context": "Ты — Agent A. Реши задачу полностью, самостоятельно. Верни рабочий код + пояснение.",
        "toolsets": ["terminal", "file"],
    },
    {
        "goal": task,
        "context": "Ты — Agent B. Реши задачу полностью, самостоятельно. Верни рабочий код + пояснение.",
        "toolsets": ["terminal", "file"],
    },
    # Agent C — опционально, если models=3
])
```

### Шаг 4 — Арбитр оценивает и выбирает

```python
arbiter_task = f"""Ты — арбитр. Оцени решения задачи и выбери лучшее.

## Задача
{task}

## Критерии оценки (приоритет — БЕЗОПАСНОСТЬ > КАЧЕСТВО)
1. Безопасность — нет инъекций, утечек, опасных операций
2. Корректность — решает ли задачу полностью
3. Производительность — оптимальный алгоритм
4. Читаемость — понятный код, комментарии
5. Надёжность — обработка ошибок, краевые случаи

## Решение A
{results[0].summary}

## Решение B
{results[1].summary}

## Решение C (если есть)
{results[2].summary if len(results) > 2 else '—'}

## Формат ответа
1. **Вердикт**: выбрано решение A / B / C / гибрид
2. **Почему**: 2-3 предложения с обоснованием
3. **Итоговое решение**: код/текст (выбранное или синтезированное)
4. **Что можно улучшить**: если есть замечания
"""

final = delegate_task(goal=arbiter_task)
```

## Режимы работы арбитра

### ⚖️ Режим «Судья» (Judge) — по умолчанию
Арбитр выбирает **одно лучшее решение** из предложенных.

```
Задача: "Напиши парсер CSV"
→ Agent A: решение 1
→ Agent B: решение 2
→ Agent C: решение 3
→ 🏆 Судья: выбрано решение B (безопаснее всех, обрабатывает edge cases)
```

**Когда использовать:** код, фикс бага, оптимизация, алгоритмы — где важно одно правильное решение.

### 🧩 Режим «Синтезатор» (Merger)
Арбитр объединяет **лучшее из каждого** решения в одно итоговое.

```
Задача: "Предложи архитектуру системы тегов"
→ Agent A: Pydantic-схемы (безопасность)
→ Agent B: SQL-запросы + кэш (производительность)
→ Agent C: сервисный слой (расширяемость)
→ 🧩 Синтезатор: берёт схемы из A, кэш из B, сервис из C
  → итог: полная архитектура
```

**Правила синтеза:**
- Не тупо склеить, а взять лучшее из каждого
- Убрать противоречия (если A и B дают разные решения одной проблемы — выбрать одно)
- Дополнить недостающее (если ни один не покрыл аспект — добавить от арбитра)

**Когда использовать:** архитектура, рефакторинг, тесты — где нужно собрать все лучшее в одном месте.

### 💡 Режим «Мозговой штурм» (Brainstorm)
Арбитр берёт **идеи из всех решений** и синтезирует **новое**, которого не было ни у одного агента.

```
Задача: "Как улучшить производительность приложения?"
→ Agent A: добавить Redis-кэш
→ Agent B: переписать SQL-запросы
→ Agent C: добавить CDN для статики
→ 💡 Мозговой штурм: "Архитектура: CDN + Redis + оптимизированные SQL +
    добавляем от себя: асинхронную очередь для тяжёлых задач"
```

**Когда использовать:** мозговой штурм, стратегия, дизайн, планирование — где рождается новое.

## Как выбрать режим

| Тип задачи | Режим | Пример |
|-----------|-------|--------|
| **Код, алгоритм** | ⚖️ Судья | «Напиши функцию X» |
| **Архитектура, рефакторинг** | 🧩 Синтезатор | «Как переписать модуль Y» |
| **Тесты** | 🧩 Синтезатор | «Напиши тесты для Z» |
| **Мозговой штурм** | 💡 Brainstorm | «Как улучшить X» |
| **Стратегия/план** | 💡 Brainstorm | «План внедрения Y» |
| **Фикс бага** | ⚖️ Судья | «Почему падает тест» |
| **Не знаешь** | 🧩 Синтезатор | Универсальный режим |

### Промпт для арбитра в каждом режиме

**Судья:**
```
Ты — ⚖️ судья. Выбери ОДНО лучшее решение из предложенных.
Критерии: безопасность > качество > производительность > читаемость.
Верни: какой вариант выбран и почему.
```

**Синтезатор:**
```
Ты — 🧩 синтезатор. Объедини ЛУЧШЕЕ ИЗ КАЖДОГО решения в одно итоговое.
Правила:
- Из A возьми лучшее, из B возьми лучшее, из C возьми лучшее
- Если решения противоречат друг другу — выбери безопаснейшее
- Если аспект не покрыт никем — добавь от себя
- Итоговое решение должно быть полным и непротиворечивым
Верни: итоговое решение + что взято из каждого источника.
```

**Мозговой штурм:**
```
Ты — 💡 фасилитатор мозгового штурма. Проанализируй все предложенные идеи.
Не выбирай и не объединяй — создай НОВОЕ решение, которого нет ни у одного агента,
но которое использует лучшие идеи из каждого.
Верни: новое решение + какие идеи откуда взяты + что добавлено от себя.
```

## Оценка времени и параллельность

### Системные лимиты

| Параметр | Значение |
|----------|----------|
| CPU | i5-3337U — 2 ядра / 4 потока |
| RAM | 8 GB DDR3 |
| Hermes `max_concurrent_children` | **3** — макс параллельных subagent'ов |
| OpenCode Zen | Работает через API, не нагружает CPU |

### Как работает параллельность в ensemble

```
3 агента → ЗАПУСКАЮТСЯ ОДНОВРЕМЕННО (parallel, лимит 3)
     ↓
1 арбитр → ЗАПУСКАЕТСЯ ПОСЛЕ (sequential)
     ↓
Итог: ~3-4 мин
```

**Почему не 4 агента:**
- `max_concurrent_children: 3` — Hermes не позволит запустить 4-й параллельно
- 4-й пришлось бы ждать, пока освободится слот — время вырастет до ~5 мин
- Тесты показали: 4-я модель редко даёт новый подход (diminishing returns)

**Почему не параллелить арбитра:**
- Арбитр должен видеть ВСЕ решения агентов, прежде чем оценивать

### Время выполнения

| Конфигурация | Параллельно | Время |
|-------------|------------|-------|
| 2 агента + арбитр | 2 сразу | ~2-3 мин |
| **3 агента + арбитр** 🔥 | **3 сразу** | **~3-4 мин ⭐** |
| 4 агента + арбитр | 3+1 (без прироста) | ~5 мин |

## Common Pitfalls

1. **Не использовать для простых задач** — оверхед 2-10 мин не оправдан для «Сколько будет 2+2?»
   > Эмпирика: на тривиальных задачах diversity = 0%, ensemble не даёт выгоды (см. `references/test-results-2026-06-16.md`)
2. **Зависимость от API** — если бесплатные модели недоступны, ensemble не сработает
3. **Одинаковые модели** — если все агенты используют одну модель, разнообразия решений не будет
4. **Арбитр — та же модель** — может иметь те же слепые пятна, что и агенты. Лучше если арбитр — другая модель.
   > Fallback DeepSeek — это компромисс: он же Agent A, но 284B против 7B MiMo — достаточный отрыв. DeepSeek самокритичен (проверено).
5. **CPU-only ноутбук** — local модели медленные. Использовать только в паре с быстрыми API-моделями
6. **Безопасность важнее количества** — если 2 агента дали безопасное решение, а 3-ий — опасное, арбитр должен отсеять опасное
7. **Nemotron-3 Ultra нестабилен** — на длинных промптах (>200 слов) через OpenCode Zen может упасть. Использовать fallback DeepSeek.
8. **Проверять модели через правильный API** — Ollama cloud модели работают через суффикс `:cloud` и локальный API (port 11434), не через `ollama run`.
